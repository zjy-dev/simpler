# InnoDB 记录结构

## InnoDB 页简介

InnoDB 以**页（Page）**为单位管理存储空间，页是磁盘与内存交互的基本单位，默认大小 **16KB**。读取数据时最少从磁盘读 16KB 到内存，刷盘时也至少写 16KB。

## COMPACT 行格式

InnoDB 支持 4 种行格式：`Compact`、`Redundant`、`Dynamic`、`Compressed`。MySQL 5.7+ 默认使用 `Dynamic`，它与 `Compact` 结构基本一致，仅行溢出处理不同。

指定行格式：

```sql
CREATE TABLE t (...) ROW_FORMAT=COMPACT;
ALTER TABLE t ROW_FORMAT=DYNAMIC;
```

一条 COMPACT 格式的记录由**额外信息**和**真实数据**两部分组成：

### 记录的额外信息

额外信息包含三部分：变长字段长度列表、NULL 值列表、记录头信息。

#### 变长字段长度列表

`VARCHAR(M)`、`VARBINARY(M)`、`TEXT`、`BLOB` 等变长类型的列，需要在记录开头存储其实际数据占用的字节数。各变长字段的长度按列顺序**逆序**排列，形成变长字段长度列表。

长度字段占用 1 字节还是 2 字节，规则如下（`W` 为字符集单字符最大字节数，`M` 为声明的字符数，`L` 为实际字节数）：

| 条件 | 长度字段占用 |
|------|-------------|
| `M × W ≤ 255` | 1 字节 |
| `M × W > 255` 且 `L ≤ 127` | 1 字节 |
| `M × W > 255` 且 `L > 127` | 2 字节 |

区分 1/2 字节的方法：用该字节的最高位作标志位，0 表示单独一个字段长度，1 表示这是半个字段长度（即用了 2 字节）。

关键细节：

- **值为 NULL 的列不存储长度**，NULL 值由 NULL 值列表处理。
- 若表中所有列都是定长类型，则不存在变长字段长度列表。
- `CHAR(M)` 在**变长字符集**（如 `utf8`）下也会被加入变长字段长度列表；在定长字符集（如 `ascii`）下不会。

![变长字段长度列表示例](images/04-03.png)

#### NULL 值列表

将允许为 NULL 的列（排除主键列和 `NOT NULL` 列）按列顺序**逆序**，每列对应一个二进制位：

- `1` → 该列值为 NULL
- `0` → 该列值不为 NULL

位数不足整字节时，高位补 0。例如表有 3 个可 NULL 列，则 NULL 值列表占 1 字节；9 个可 NULL 列则占 2 字节。

若表中没有允许 NULL 的列，则 NULL 值列表不存在。

![NULL值列表示例](images/04-08.png)

#### 记录头信息

固定占用 **5 字节（40 bit）**，各字段含义：

| 名称 | 大小(bit) | 描述 |
|------|----------|------|
| `预留位1` | 1 | 未使用 |
| `预留位2` | 1 | 未使用 |
| `delete_mask` | 1 | 标记记录是否被删除 |
| `min_rec_mask` | 1 | B+ 树每层非叶子节点的最小记录标记 |
| `n_owned` | 4 | 当前记录拥有的记录数（页目录相关） |
| `heap_no` | 13 | 当前记录在记录堆中的位置 |
| `record_type` | 3 | 0=普通记录, 1=B+树非叶节点, 2=最小记录, 3=最大记录 |
| `next_record` | 16 | 下一条记录的相对位置（形成单链表） |

![记录头信息](images/04-09.png)

### 记录的真实数据

真实数据部分存储用户定义的各列数据。

`CHAR(M)` 在定长字符集下固定占用 `M` 字节，不足部分用空格（`0x20`）填充。在变长字符集下至少占用 `M` 字节（避免原地更新时产生碎片），最多占用 `M × W` 字节。

值为 NULL 的列不在真实数据区占用空间，已通过 NULL 值列表标记。

![真实数据示例](images/04-11.png)

### 隐藏列

InnoDB 为每条记录自动添加隐藏列：

| 列名 | 是否必须 | 大小 | 描述 |
|------|---------|------|------|
| `DB_ROW_ID` | 否 | 6 字节 | 行 ID，无主键且无唯一索引时自动生成作为隐式主键 |
| `DB_TRX_ID` | 是 | 6 字节 | 事务 ID，标识最后修改该记录的事务 |
| `DB_ROLL_PTR` | 是 | 7 字节 | 回滚指针，指向 undo log 中该记录的上一个版本 |

主键生成策略：用户定义主键 → `UNIQUE` 键 → 自动添加 `DB_ROW_ID`。`DB_TRX_ID` 和 `DB_ROLL_PTR` 始终存在，是 MVCC 机制的基础。

## 行溢出

一个页 16KB（16384 字节），而 `VARCHAR(M)` 最多能存 65532 字节（加上 2 字节长度 + 1 字节 NULL 标识 = 65535 上限）。当单条记录数据量超过页容量时，就会发生**行溢出**。

MySQL 规定一个页中至少存放两行记录。对于单列表，行溢出的临界点约为 **8098 字节**（由公式 `136 + 2 × (27 + n) > 16384` 推导）。多列表临界点更低。

COMPACT/Redundant 格式的溢出处理：在本页存储前 **768 字节** + 20 字节的溢出页地址指针，剩余数据存入溢出页。

![行溢出示意](images/04-17.png)

不只是 `VARCHAR`，`TEXT` 和 `BLOB` 类型数据量过大时同样会行溢出。

## Dynamic 与 Compressed 行格式

`Dynamic`（MySQL 5.7+ 默认）和 `Compressed` 与 `Compact` 结构基本一致，区别在于行溢出处理方式：

| 行格式 | 溢出处理 |
|--------|---------|
| Compact / Redundant | 本页保留前 768 字节 + 20 字节溢出页指针 |
| Dynamic / Compressed | 本页**仅存 20 字节溢出页指针**，所有数据存入溢出页 |

`Compressed` 额外会对页面数据进行压缩以节省磁盘空间。
