# InnoDB 数据页结构

InnoDB 以**页**为单位管理存储空间，默认页大小 `16KB`。存放表中行记录的页称为**数据页**（类型 `FIL_PAGE_INDEX`，即索引页）。

## 数据页的七个组成部分

一个数据页的 `16KB` 空间划分为 7 个部分：

| 名称 | 大小 | 说明 |
|------|------|------|
| `File Header` | 38 字节 | 页的通用信息：页号、校验和、前后页指针等 |
| `Page Header` | 56 字节 | 数据页专有状态：记录数、槽数、空闲空间位置等 |
| `Infimum + Supremum` | 26 字节 | 两条虚拟记录，分别代表页内最小和最大记录 |
| `User Records` | 不固定 | 实际存储的行记录 |
| `Free Space` | 不固定 | 尚未使用的空间 |
| `Page Directory` | 不固定 | 由若干**槽**（Slot）组成，存储分组中最大记录的地址偏移量 |
| `File Trailer` | 8 字节 | 校验和 + LSN，用于检测页写入是否完整 |

插入记录时，从 `Free Space` 中划出一块空间分配给 `User Records`。当 `Free Space` 耗尽，需要申请新的数据页。

## 记录在页中的存储

### 记录头信息

每条记录都有 5 字节的**记录头信息**，核心字段：

| 字段 | 位数 | 说明 |
|------|------|------|
| `delete_mask` | 1 | 删除标记。1 表示已删除，记录仍留在页中，空间可复用 |
| `min_rec_mask` | 1 | B+ 树每层非叶子节点的最小记录会标记为 1 |
| `n_owned` | 4 | 该记录所在分组的记录数（仅组内最大记录维护此值） |
| `heap_no` | 13 | 记录在本页中的堆位置编号 |
| `record_type` | 3 | 0=普通记录，1=B+ 树非叶节点记录，2=最小记录，3=最大记录 |
| `next_record` | 16 | 指向下一条记录真实数据的**地址偏移量** |

### Infimum 和 Supremum 记录

每个数据页自动包含两条**虚拟记录**：

- **Infimum**（最小记录）：`heap_no = 0`，`record_type = 2`
- **Supremum**（最大记录）：`heap_no = 1`，`record_type = 3`

这两条记录存放在 `Infimum + Supremum` 区域，不在 `User Records` 中。它们各由 5 字节记录头信息 + 8 字节固定数据组成。无论插入多少用户记录，Infimum 始终最小，Supremum 始终最大。

![](images/05-06.png)

### 记录的排列（next_record 单链表）

页中所有记录通过 `next_record` 按**主键值从小到大**串成单链表：

```
Infimum → 主键最小的用户记录 → ... → 主键最大的用户记录 → Supremum
```

关键规则：

- `next_record` 是从当前记录真实数据到下一条记录真实数据的偏移量（向左读头信息，向右读数据）
- 链表顺序与插入顺序无关，始终按主键排序
- Supremum 的 `next_record = 0`，表示链表终止

![](images/05-08.png)

### 删除记录的处理

删除一条记录时：

1. 将该记录的 `delete_mask` 置为 1，**不回收存储空间**
2. 该记录的 `next_record` 置为 0
3. 前一条记录的 `next_record` 指向后一条记录，链表保持连通
4. 所在组的 `n_owned` 值减 1

被删除的记录通过 `next_record` 组成**垃圾链表**，其空间可被后续新记录复用。

![](images/05-09.png)

## Page Directory（页目录）

单链表只能顺序遍历，记录多时效率低。Page Directory 通过分组 + 二分查找加速检索。

### 分组规则

1. 将所有**未删除的记录**（含 Infimum 和 Supremum）分成若干组
2. 每组的**最后一条记录**（组内主键最大者）的 `n_owned` 记录该组的记录条数
3. 将每组最后一条记录的**地址偏移量**提取出来，按序存入 `Page Directory`，每个偏移量称为一个**槽**（Slot）

各组记录数的约束：

| 分组 | 允许的记录条数 |
|------|---------------|
| Infimum 所在组 | 恰好 1 条 |
| Supremum 所在组 | 1 ~ 8 条 |
| 其余中间组 | 4 ~ 8 条 |

分组的动态维护过程：

1. 初始状态只有 Infimum 和 Supremum 两条记录，分属两个组
2. 每插入一条记录，找到主键值刚好大于它的槽，将该槽对应记录的 `n_owned` 加 1
3. 当某组记录数达到 8 条后再插入时，拆分为 4 + 5 两组，Page Directory 新增一个槽

![](images/05-14.png)

### 查找过程（二分查找）

在数据页中按主键查找记录分两步：

**第一步：二分法定位槽**

槽从 0 到 n 编号，对应的主键值从小到大排列。设 `low = 0`，`high = n`：

1. 取中间槽 `mid = (low + high) / 2`
2. 比较 `mid` 槽对应记录的主键值与目标值
3. 目标值大，则 `low = mid`；目标值小，则 `high = mid`
4. 当 `high - low = 1` 时，目标记录在 `high` 槽对应的组中

**第二步：链表遍历组内记录**

通过 `low` 槽对应记录的 `next_record` 找到 `high` 槽所在组的第一条记录，沿链表顺序遍历（最多 8 次），找到目标记录。

**示例**：查找主键值为 6 的记录，假设有 5 个槽（0~4）：

```
1. mid = (0+4)/2 = 2，槽2主键=8 > 6 → high=2
2. mid = (0+2)/2 = 1，槽1主键=4 < 6 → low=1
3. high - low = 1，目标在槽2对应的组中
4. 取槽1记录(主键4)的 next_record → 主键5 → 主键6，找到
```

## Page Header（页面头部）

56 字节，存储数据页的状态信息：

| 字段 | 大小 | 说明 |
|------|------|------|
| `PAGE_N_DIR_SLOTS` | 2 字节 | Page Directory 中的槽数量 |
| `PAGE_HEAP_TOP` | 2 字节 | Free Space 的起始地址 |
| `PAGE_N_HEAP` | 2 字节 | 页中所有记录数（含 Infimum/Supremum 和已删除记录） |
| `PAGE_FREE` | 2 字节 | 垃圾链表的头节点地址 |
| `PAGE_GARBAGE` | 2 字节 | 已删除记录占用的总字节数 |
| `PAGE_LAST_INSERT` | 2 字节 | 最后一次插入记录的位置 |
| `PAGE_DIRECTION` | 2 字节 | 最后一次插入的方向（左/右，即主键比上一条小/大） |
| `PAGE_N_DIRECTION` | 2 字节 | 同一方向连续插入的记录数 |
| `PAGE_N_RECS` | 2 字节 | 有效用户记录数（不含 Infimum/Supremum 和已删除记录） |
| `PAGE_MAX_TRX_ID` | 8 字节 | 修改该页的最大事务 ID（仅二级索引页有意义） |
| `PAGE_LEVEL` | 2 字节 | 该页在 B+ 树中的层级（叶子节点为 0） |
| `PAGE_INDEX_ID` | 8 字节 | 该页所属索引的 ID |
| `PAGE_BTR_SEG_LEAF` | 10 字节 | B+ 树叶子段头信息（仅 Root 页） |
| `PAGE_BTR_SEG_TOP` | 10 字节 | B+ 树非叶子段头信息（仅 Root 页） |

## File Header（文件头部）

38 字节，所有类型的页通用：

| 字段 | 大小 | 说明 |
|------|------|------|
| `FIL_PAGE_SPACE_OR_CHKSUM` | 4 字节 | 页的校验和 |
| `FIL_PAGE_OFFSET` | 4 字节 | 页号，唯一标识一个页 |
| `FIL_PAGE_PREV` | 4 字节 | 上一个页的页号 |
| `FIL_PAGE_NEXT` | 4 字节 | 下一个页的页号 |
| `FIL_PAGE_LSN` | 8 字节 | 页面最后修改时的日志序列号（LSN） |
| `FIL_PAGE_TYPE` | 2 字节 | 页类型（数据页为 `0x45BF`） |
| `FIL_PAGE_FILE_FLUSH_LSN` | 8 字节 | 仅系统表空间使用，文件至少刷新到的 LSN |
| `FIL_PAGE_ARCH_LOG_NO_OR_SPACE_ID` | 4 字节 | 所属表空间 ID |

通过 `FIL_PAGE_PREV` 和 `FIL_PAGE_NEXT`，所有数据页组成**双向链表**，物理上不必连续存储。

![](images/05-15.png)

## File Trailer（文件尾部）

8 字节，用于校验页的写入完整性：

- **前 4 字节**：校验和，与 `File Header` 中的校验和对应。页在内存中修改后先写头部校验和，全部写完后写尾部校验和。若二者不一致，说明写入过程中发生了中断（如断电）。
- **后 4 字节**：页面最后修改时的 LSN 低 4 字节，同样用于完整性校验。

## 面试要点速记

1. 数据页 `16KB`，分 7 个部分；记录通过 `next_record` 按主键排序形成**单链表**
2. 删除记录不回收空间，标记 `delete_mask = 1`，加入垃圾链表待复用
3. Page Directory 将记录分组，每组 1~8 条，每个槽指向组内最大记录
4. 页内按主键查找：先**二分查找**定位槽，再**遍历组内链表**（最多 8 次比较）
5. 数据页之间通过 `FIL_PAGE_PREV / NEXT` 组成**双向链表**
6. `File Trailer` 与 `File Header` 的校验和配合检测页写入完整性
