# B+ 树索引

InnoDB 的数据页通过双向链表串联，页内记录按主键组成单向链表，每页有页目录（Page Directory）支持二分查找。索引就是在此基础上构建的多级目录结构。

![页与记录的关系](images/06-01.png)

## 没有索引时的查找

**页内查找**分两种情况：

- **主键搜索**：利用页目录二分定位槽，再遍历槽内分组，速度快。
- **非主键搜索**：没有对应的页目录，只能从最小记录开始逐条遍历单链表，效率极低。

**跨页查找**需要两步：先定位记录所在页，再在页内查找。没有索引时，无论按什么列搜索，都只能沿双向链表从第一页逐页扫描——这就是全表扫描，数据量大时极慢。

## 索引的本质

### 页内目录 → 页间目录

每个数据页内部已有页目录，可以二分查主键。问题在于：**页与页之间没有这样的快速定位机制**。

为解决跨页查找，需要为所有数据页建立目录项，每个目录项包含：

- `key`：该页中用户记录的最小主键值
- `page_no`：页号

![简易目录示意](images/06-09.png)

有了目录，查主键 20 的记录只需：先在目录项中二分定位到目标页（12 < 20 < 209 → 页 9），再在页 9 内二分查找。这就是**索引**的雏形。

### 从目录到 B+ 树

上述方案要求目录项在物理上连续存储，存在两个问题：

1. 记录数量增长后，目录项需要的连续空间可能超过单个 16KB 页。
2. 删除页后需要移动后续所有目录项，代价高。

InnoDB 的解决方式：**用数据页来存储目录项**。目录项记录和用户记录结构相同，通过 `record_type` 区分：

| record_type | 含义 |
|:-----------:|------|
| 0 | 普通用户记录 |
| 1 | 目录项记录 |
| 2 | 最小记录 |
| 3 | 最大记录 |

![目录项记录存储在数据页中](images/06-10.png)

目录项记录只包含**主键值 + 页号**两个字段，比用户记录小得多，但一个页容量有限。当目录项记录超出单页容量时，就再分配新的目录项页。

多个目录项页之上，再建一层更高级的目录页来索引它们——层层嵌套，最终形成多级树形结构：

![B+ 树结构](images/06-13.png)

这就是 **B+ 树**。

- **叶子节点**（最底层，第 0 层）：存放实际用户记录。
- **内节点**（非叶子节点）：存放目录项记录。
- **根节点**：最顶层的节点。

### B+ 树的存储容量

假设叶子节点每页存 100 条用户记录，内节点每页存 1000 条目录项记录：

| 层数 | 最大记录数 |
|:----:|----------:|
| 1 | 100 |
| 2 | 10 万 |
| 3 | 1 亿 |
| 4 | 1000 亿 |

实际使用中 B+ 树一般不超过 4 层。查找一条记录最多 4 次页内二分查找，效率极高。

## 聚簇索引

上面构建的 B+ 树就是聚簇索引（Clustered Index）。它有两个核心特征：

**1. 按主键排序组织所有记录和页：**

- 页内记录按主键排成单向链表。
- 叶子节点（用户记录页）按主键排成双向链表。
- 同层的内节点页也按主键排成双向链表。

**2. 叶子节点存储完整的用户记录**（包括所有列和隐藏列）。

关键要点：

- InnoDB **自动创建**聚簇索引，无需手动建。
- 聚簇索引即数据存储方式本身——**索引即数据，数据即索引**。
- 每张 InnoDB 表有且仅有一个聚簇索引。

## 二级索引（辅助索引）

聚簇索引只能加速主键查询。若要按其他列查找，需要额外建 B+ 树。

以 `c2` 列建索引为例：

![c2 列的二级索引](images/06-14.png)

与聚簇索引的区别：

| 对比项 | 聚簇索引 | 二级索引 |
|--------|----------|----------|
| 排序依据 | 主键 | 索引列（如 c2） |
| 叶子节点存储 | 完整用户记录 | 索引列值 + 主键值 |
| 目录项记录 | 主键 + 页号 | 索引列 + 主键 + 页号 |
| 查完整记录 | 一次查找即可 | 需要回表 |

### 回表过程

用 `c2 = 4` 查找完整记录的步骤：

1. 在二级索引 B+ 树中，从根节点逐层定位到叶子节点。
2. 在叶子节点找到匹配记录，取出其中存储的主键值。
3. **拿主键值回到聚簇索引中再查一次**，取出完整用户记录。

第 3 步就是**回表**。一次非主键查询需要遍历两棵 B+ 树。

为什么不在二级索引叶子节点直接存完整记录？因为每建一个索引就要复制一份全部数据，太浪费空间。

## 联合索引

同时以多个列作为排序规则建立的 B+ 树。例如为 `(c2, c3)` 建立联合索引：

- 先按 `c2` 排序。
- `c2` 相同时，再按 `c3` 排序。

![联合索引 (c2, c3)](images/06-15.png)

联合索引的结构：

- 目录项记录：`c2 + c3 + 页号`
- 叶子节点记录：`c2 + c3 + 主键 c1`

**最左前缀原则：** 联合索引 `(c2, c3)` 能加速以下查询：

- `WHERE c2 = x`（用到 c2 排序）
- `WHERE c2 = x AND c3 = y`（同时用到 c2、c3 排序）

但**不能**单独加速 `WHERE c3 = y`，因为在 c2 不确定时 c3 并不是全局有序的。

**联合索引 vs 多个单列索引：**

- `INDEX(c2, c3)`：建 1 棵 B+ 树。
- `INDEX(c2)` + `INDEX(c3)`：建 2 棵 B+ 树。

两者的索引结构和适用查询完全不同。

## InnoDB 中 B+ 树索引的注意事项

### 根页面固定不动

B+ 树的形成过程：

1. 创建索引时分配一个根节点页面，初始为空。
2. 插入记录时先存入根节点。
3. 根节点满后，将所有记录复制到新页 a，再分裂出新页 b，根节点升级为存储目录项记录的内节点。

核心规则：**根节点页面自创建后不再移动**。InnoDB 将根节点页号记录在数据字典中，通过固定入口访问整棵 B+ 树。

### 内节点目录项的唯一性

二级索引的内节点如果只存 `索引列 + 页号`，当索引列有大量重复值时，无法判断新记录该插入哪个子页。

![只有索引列+页号时的歧义](images/06-16.png)

因此，二级索引内节点的目录项记录实际包含三部分：

- 索引列的值
- **主键值**
- 页号

加入主键后保证同层内目录项记录的唯一性，消除定位歧义。

![加入主键后的二级索引内节点](images/06-17.png)

### 每个页面至少存 2 条记录

B+ 树高效的前提是每层能过滤大量子目录。如果每页只存 1 条记录，树的层级会急剧增长，退化为链表。因此 InnoDB 要求每个数据页至少存放 2 条记录。

## InnoDB vs MyISAM 索引对比

| 对比项 | InnoDB | MyISAM |
|--------|--------|--------|
| 数据存储 | 聚簇索引叶子节点 | 独立数据文件，按插入顺序 |
| 主键索引叶子 | 完整用户记录 | 主键 + 行号 |
| 二级索引叶子 | 索引列 + 主键 | 索引列 + 行号 |
| 聚簇索引 | 有，且唯一 | 无，所有索引都是二级索引 |
| 回表方式 | 主键 → 聚簇索引 | 行号 → 数据文件 |

核心区别：InnoDB 索引即数据；MyISAM 索引与数据分离。

## 创建和删除索引

```sql
-- 建表时创建联合索引
CREATE TABLE t (
    id INT PRIMARY KEY,
    a INT, b CHAR(1),
    INDEX idx_a_b (a, b)
);

-- 添加/删除索引
ALTER TABLE t ADD INDEX idx_a (a);
ALTER TABLE t DROP INDEX idx_a;
```

InnoDB 自动为主键和 `UNIQUE` 列创建索引，其他列需手动创建。每多一个索引就多一棵 B+ 树，写入时需维护所有索引排序，不宜盲目建索引。
