# 受限直接执行

OS 需要解决两个问题：**高效**（不能什么都自己来）和 **控制**（不能让进程为所欲为）。受限直接执行（Limited Direct Execution，LDE）是解决方案。

## 用户模式与内核模式

硬件提供两种运行模式：

- **用户模式**（user mode）：进程在这里运行，不能直接执行特权操作（如 I/O、修改页表）
- **内核模式**（kernel mode）：OS 在这里运行，什么都能干

进程想做特权操作时必须通过系统调用。

## 系统调用与 trap

系统调用的本质是一条特殊指令——**trap**。执行 trap 时：

1. 硬件把寄存器保存到该进程的**内核栈**
2. 跳转到预设的 **trap 处理程序**
3. 切换到内核模式
4. OS 执行请求的操作
5. 执行 **return-from-trap**，恢复寄存器，回到用户模式继续执行

## trap 表

OS 在启动时设置 **trap 表**（trap table），告诉硬件各种异常和系统调用对应的处理程序地址。这是在内核模式下做的——用户程序不能修改 trap 表，否则就能跳到任意内核代码了。

每个系统调用有一个**系统调用号**，用户程序设置好调用号后执行 trap，OS 查表找到对应处理函数执行。这样用户程序不能直接指定跳转地址，只能请求已注册的服务。

## 进程切换

OS 怎么从一个正在跑的进程手里拿回 CPU？靠 **时钟中断**（timer interrupt）。

时钟设备每隔几毫秒产生一次中断，当前进程被打断，OS 的中断处理程序开始执行。此时 OS 可以决定：继续跑当前进程，还是切换到别的进程。

## 上下文切换

如果 OS 决定切换进程，执行**上下文切换**（context switch）。注意这里有两次寄存器保存/恢复：

1. **硬件**：trap 时自动把用户寄存器存到当前进程的内核栈
2. **OS**：把当前进程的内核寄存器存到它的 PCB，再从目标进程的 PCB 恢复内核寄存器

切换到目标进程的内核栈后，执行 return-from-trap 就回到了目标进程的用户空间。

以 xv6 的 `swtch()` 为例，它只保存/恢复被调用者保存的寄存器（callee-saved），因为调用者保存的寄存器已经由调用约定处理了。

## LDE 协议总结

| 阶段 | 操作 |
|------|------|
| 启动 | OS 初始化 trap 表，启动时钟中断 |
| 运行进程 | OS 设好寄存器，return-from-trap 到用户模式 |
| 系统调用 | trap → 内核处理 → return-from-trap |
| 时钟中断 | 中断 → OS 决定是否上下文切换 → return-from-trap 到目标进程 |
