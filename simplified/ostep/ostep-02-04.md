# 基于锁的并发数据结构

给数据结构加锁的方式很影响性能。粗粒度锁（一把大锁保护整个结构）简单但扩展性差；细粒度锁复杂但并发度高。

## 并发计数器

### 简单方案

一把锁保护一个全局计数器。正确但完全串行，线程越多越慢。

### 懒惰计数器（sloppy counter）

每个 CPU 一个**本地计数器**（带各自的锁），加上一个**全局计数器**（带全局锁）。

- 线程在自己 CPU 的本地计数器上操作——不同 CPU 之间无竞争
- 本地计数器累积到阈值 S 后，一次性转移到全局计数器，本地清零

S 越小，全局值越精确但转移越频繁（像精确计数器一样慢）。S 越大，转移越少但全局值越不精确。S = 1024 时，4 个 CPU 的性能已经接近无锁单线程。

## 并发链表

最直接的做法是整个链表一把锁。性能一般但够用。

Hand-over-hand locking（每个节点一把锁，遍历时逐步交接）理论上并发度高，但加锁/解锁的开销太大，实际很少比一把大锁快。

## 并发队列（Michael-Scott 队列）

头部一把锁，尾部一把锁，入队和出队互不阻塞。

关键技巧：用一个**哨兵节点**（dummy node）分开头尾操作。入队操作只碰尾部，出队操作只碰头部。没有哨兵的话，队列只有一个元素时头尾指向同一个节点，两把锁就冲突了。

## 并发哈希表

最简单也最实用的方案：**每个桶一把锁**（per-bucket lock）。不同桶的操作完全并行。

扩展性好。线程数增加时吞吐近乎线性增长——因为大部分操作落在不同的桶里。这是实际系统中最常见的并发哈希表设计。
