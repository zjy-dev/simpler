# 多处理器调度

单 CPU 的调度已经不够，现代机器都是多核的。多处理器调度要面对新的硬件问题。

## 缓存与一致性

每个 CPU 都有自己的缓存（L1/L2，有些 L3 共享）。同一份数据可能在多个 CPU 的缓存中各有一份，某个 CPU 修改数据后，其他 CPU 缓存里的旧值就不对了。

硬件通过**总线窥探**（bus snooping）解决这个问题：每个缓存监控总线上的内存操作，发现有人写了自己缓存中存在的数据时，立即将自己的副本标记为无效或更新。这对上层软件基本透明。

## 缓存亲和性

一个进程在某个 CPU 上跑过之后，该 CPU 的缓存里已经有它的数据了。如果下次调度到同一个 CPU，缓存命中率高，性能好。调度器应尽量让进程保持在同一个 CPU 上，这叫**缓存亲和性**（cache affinity）。

## 两种调度方案

### SQMS（单队列多处理器调度）

所有进程放在一个全局队列里，每个 CPU 从里面取。

- 优点：简单，负载自动均衡
- 缺点：全局队列需要加锁，CPU 越多竞争越激烈，扩展性差。缓存亲和性也难保证（进程可能被随机分到不同 CPU）

### MQMS（多队列多处理器调度）

每个 CPU 有自己的队列，进程创建时分配到某个队列。

- 优点：无需全局锁，扩展性好。天然有缓存亲和性（进程固定在一个 CPU）
- 缺点：可能负载不均。如果某个 CPU 的进程都结束了，它就空转了

解决负载不均的办法是**工作窃取**（work stealing）：空闲 CPU 偶尔看一眼别的队列，如果人家进程多就偷几个过来。检查频率需要权衡——太频繁增加开销，太少则负载失衡时间长。

## Linux 调度器

Linux 内核里同时存在多种调度器：

- **CFS**（Completely Fair Scheduler）：当前默认，用红黑树按虚拟运行时间排序
- **BFS**（Brain Fuck Scheduler）：单队列设计，用最早合适截止时间（EEVDF）思路
- **O(1) 调度器**：老版本，多队列，基于优先级

不同调度器对不同工作负载各有优劣。
