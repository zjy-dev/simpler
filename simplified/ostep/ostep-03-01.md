# I/O 设备

## 系统架构

CPU 和内存通过**内存总线**连接（最快）。显卡等高性能设备接在 **I/O 总线**（如 PCIe）上。磁盘、鼠标、键盘等慢设备接在**外围总线**（如 USB、SATA）上。

越快的总线越短、越贵、设备越少。层次结构平衡了速度和成本。

## 设备接口

一个标准设备对外暴露三个寄存器：

- **状态寄存器**：读取设备当前状态
- **命令寄存器**：告诉设备执行什么操作
- **数据寄存器**：传输数据

OS 通过读写这些寄存器与设备交互。

## 与设备交互的方式

**显式 I/O 指令**（x86 的 `in`/`out`）：特权指令，直接指定设备端口号读写。

**内存映射 I/O**（memory-mapped I/O）：设备寄存器被映射到物理地址空间的某个区域，OS 直接用 load/store 指令访问。

## 轮询 vs 中断

**轮询**（polling）：CPU 反复读状态寄存器，等设备就绪。简单粗暴，浪费 CPU。

**中断**（interrupt）：CPU 发出请求后去干别的事，设备完成后触发中断，OS 的中断处理程序被唤起。CPU 利用率高。

但中断不是万能的：

- 设备很快时，轮询可能比中断好——省了上下文切换的开销
- 网络流量大时中断太频繁（**活锁**——CPU 全在处理中断，没空干活）→ 混合方案：先轮询一会儿，没结果再转中断
- 中断**合并**（coalescing）：设备攒一批完成通知再触发一次中断，减少中断频率

## DMA

没有 DMA 时，CPU 要一个字节一个字节地把数据搬到设备（PIO）。有了 **DMA**（Direct Memory Access），CPU 只需告诉 DMA 引擎：从内存地址 X 搬 N 字节到设备，然后 CPU 去忙别的。DMA 引擎独立完成数据传输，完事后中断通知 CPU。

## 设备驱动

OS 不可能为每种设备写专门代码。通用做法：定义一套抽象接口（如块设备接口：read block / write block），**设备驱动程序**负责把抽象接口翻译成具体设备的操作。

Linux 内核中设备驱动代码占 70% 以上，也是 bug 的重灾区。

文件系统不关心底下是 SCSI 盘还是 SATA 盘——它只对通用块层说"读第 X 块"，块层再交给对应的驱动处理。
