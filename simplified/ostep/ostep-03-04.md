# 文件系统实现

以 VSFS（Very Simple File System）为模型，理解 ext2/ext3 的核心设计。

## 磁盘布局

一个文件系统把磁盘分成若干区域：

| 区域 | 内容 |
|------|------|
| 超级块（superblock） | 文件系统元信息：inode 数量、数据块数量、各区域起始位置、魔数等 |
| inode 位图 | 每个 bit 标记对应 inode 是否被占用 |
| 数据位图 | 每个 bit 标记对应数据块是否被占用 |
| inode 表 | 存放所有 inode，每个 inode 固定大小（通常 256B） |
| 数据区 | 存放实际文件数据和目录内容 |

## inode

每个文件一个 inode，包含：

- 文件类型（普通文件、目录、符号链接等）
- 大小、权限、拥有者
- 时间戳（创建、修改、访问）
- **数据块指针**

### 多级索引

如何在 inode 里记录文件数据存在哪些块？

- **直接指针**：12 个，每个直接指向一个数据块（4KB × 12 = 48KB）
- **一级间接指针**：指向一个块，这个块里全是指针（4KB/4B = 1024 个指针 → 额外 4MB）
- **二级间接指针**：指向一个块，里面的指针指向间接块（1024 × 1024 → 额外 4GB）
- **三级间接指针**：再套一层（1024³ → 额外 4TB）

大多数文件很小（平均 2KB 左右），12 个直接指针就够了。只有大文件才需要间接指针。这种不对称设计很聪明——小文件没有间接块的开销。

另一种方案是 **extent**（指针 + 长度）：记录"从块 X 开始连续 N 块"。连续分配的大文件只需要一个 extent。ext4 和 XFS 用 extent。

## 目录

目录的数据块里存的是条目列表，每个条目包含：

- inode 号
- 条目长度（reclen）
- 文件名长度
- 文件名

删除文件时不会真的移除条目，只是把 reclen 增大（让前一个条目覆盖它）。

## 空闲空间管理

inode 位图和数据位图各用 1 bit 标记占用/空闲。分配时扫描位图找到空闲位。

ext2/3 等系统会预分配（比如一次分 8 个连续块），让文件数据尽量连续，减少碎片。XFS 用 B 树管理空闲空间。

## 读文件的路径

读 `/foo/bar` 需要的 I/O 操作：

1. 读根目录 inode（inode 2）
2. 读根目录数据块，找到 `foo` 对应的 inode 号
3. 读 `foo` 的 inode
4. 读 `foo` 的数据块，找到 `bar` 对应的 inode 号
5. 读 `bar` 的 inode
6. 读 `bar` 的数据块

路径每深一层，至少多两次 I/O（读 inode + 读目录数据）。路径越长开销越大。

## 写文件的路径

写入比读取贵得多。创建一个新文件大约需要 10 次 I/O：

- 读 inode 位图 → 分配 inode → 写 inode 位图
- 写新 inode
- 读目录数据 → 写目录数据（添加新条目）→ 读写目录 inode
- 每写一个数据块：读数据位图 → 写数据位图 → 写数据块 → 更新 inode

## 缓存

频繁读磁盘太慢。现代系统用**统一页缓存**（unified page cache），把文件数据和元数据都缓存在内存里——文件系统缓存和虚拟内存系统用同一个缓存池，动态分配。

写操作先写到缓存（**write buffering**），积攒一批后集中写盘（通常延迟 5~30 秒）。好处：

- 批量写入可以调度，减少磁盘寻道
- 有些数据写完很快又被覆盖或删除，直接省掉了磁盘写入
- 降低了单次写操作的感知延迟

代价：断电时缓存中未落盘的数据会丢失。需要 journaling 等机制保证一致性（下一章内容）。
