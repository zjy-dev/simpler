# 常见并发 Bug

并发 bug 分两大类：非死锁 bug 和死锁 bug。实际系统中非死锁 bug 占了大多数（约 97%）。

## 非死锁 Bug

### 原子性违反

"我以为这段代码会原子执行，但没有。"

MySQL 的例子：

```c
// 线程 1
if (thd->proc_info) {
    fputs(thd->proc_info, ...);
}

// 线程 2
thd->proc_info = NULL;
```

线程 1 检查非空后、使用前，线程 2 可能把它置为 NULL → 空指针崩溃。

修复：用锁把检查和使用包成一个原子操作。

### 顺序违反

"我以为线程 A 一定在线程 B 之前执行某个操作。"

比如 `mThread` 在线程 1 里创建，线程 2 直接用它——但线程 2 可能先跑到。

修复：用条件变量强制顺序。

## 死锁

两个或更多线程互相等待对方持有的资源，谁也走不了。

### 4 个必要条件

死锁发生当且仅当以下四个条件**同时**成立：

1. **互斥**：资源不能共享
2. **持有并等待**：线程持有一个资源的同时等待另一个
3. **非抢占**：资源不能被强制收回
4. **循环等待**：A 等 B，B 等 A（或更长的环）

打破任何一个条件就能避免死锁。

### 预防策略

**打破循环等待——锁排序**

给所有锁定一个全局顺序，所有线程按相同顺序获取锁。比如总是先拿 L1 再拿 L2。

如果不知道两把锁谁先谁后（比如参数传入），可以用地址排序：地址小的先拿。

这是最实用的策略。Linux 内核的锁文档里就规定了锁的获取顺序。

**打破持有并等待**

在获取任何锁之前，先拿一把全局的 prevention 锁。这样要么一次拿到所有需要的锁，要么一个都没拿。

缺点：需要预先知道要用哪些锁，而且把所有获取操作串行化了。

**打破非抢占——trylock**

用 `pthread_mutex_trylock()`：拿到了返回 0，拿不到立即返回错误（不阻塞）。

```c
top:
    lock(L1);
    if (trylock(L2) != 0) {
        unlock(L1);
        goto top;    // 释放后重试
    }
```

问题：两个线程可能反复同时获取/释放 → **活锁**（livelock）。加随机延迟能缓解。

**打破互斥——无锁编程**

用 CAS 等原子操作代替锁：

```c
void AtomicIncrement(int *val, int amount) {
    do {
        int old = *val;
    } while (CompareAndSwap(val, old, old + amount) != old);
}
```

更复杂的例子：无锁链表插入用 CAS 保证只有一个线程成功插入头部，失败的线程重试。

无锁编程避免了死锁，但编写困难，而且会遇到 ABA 问题（值被改成别的又改回来，CAS 检测不到）和活锁。

### 其他方法

**死锁避免**（Dijkstra 银行家算法）：调度器根据线程的资源需求规划执行顺序，避开死锁状态。实际系统基本不用——需要预知所有资源需求，并发度也严重受限。

**检测与恢复**：允许死锁发生，定期检测资源图中的环，发现后杀进程或回滚。数据库系统常用这个方法。
