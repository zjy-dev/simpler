# 进程 API（UNIX）

UNIX 用 `fork()`、`exec()`、`wait()` 三个系统调用来创建和管理进程。这套设计看起来奇怪，组合起来却非常强大。

## fork()

`fork()` 创建调用进程的一个几乎完全相同的副本。调用一次，返回两次：

- 父进程收到子进程的 PID（正整数）
- 子进程收到 0

```c
int rc = fork();
if (rc == 0) {
    // 子进程走这里
} else {
    // 父进程走这里，rc 是子进程 PID
}
```

子进程从 `fork()` 返回处继续执行，不是从头开始。父子谁先运行取决于调度器，不确定。

## wait()

`wait()` 让父进程等子进程结束。如果父进程先调度到，`wait()` 会阻塞直到子进程退出；如果子进程先跑完，`wait()` 立即返回。这保证了确定性的执行顺序。

## exec()

`exec()` 用一个全新的程序替换当前进程的代码和数据。调用成功后不会返回——因为当前程序已经被替换了。

```c
// 子进程里调用 exec
execvp("wc", args);  // 把自己替换成 wc 程序
// 下面这行永远不会执行
```

## 为什么要分离 fork 和 exec

分成两步是关键设计。在 `fork()` 之后、`exec()` 之前，子进程可以做任意操作——最常见的就是 **I/O 重定向**：

```c
int rc = fork();
if (rc == 0) {
    close(STDOUT_FILENO);              // 关掉标准输出
    open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);  // fd 1 指向文件
    execvp("wc", args);               // wc 的输出就写到文件了
}
```

`open()` 会分配最小的可用文件描述符。关掉 stdout（fd 1）后再 `open()`，新文件自然拿到 fd 1。`wc` 完全不知道输出被重定向了。

## shell 的工作原理

shell 本质就是个循环：

1. 打印提示符
2. 读取用户输入的命令
3. `fork()` 出子进程
4. 子进程 `exec()` 执行命令
5. 父进程 `wait()` 等子进程结束
6. 回到步骤 1

## pipe()

`pipe()` 系统调用把一个进程的输出连接到另一个进程的输入。shell 里的 `|` 就靠它实现：`grep foo file | wc -l` 把 `grep` 的 stdout 接到 `wc` 的 stdin。

## 其他 API

- `kill()`：给进程发信号（如 `SIGINT`、`SIGTSTP`）
- 信号的处理：进程可以用 `signal()` 自定义信号处理函数
