# 页替换策略

物理内存不够时需要换出页面，选谁换出是**页替换策略**的工作。选错了代价极大——磁盘访问比内存慢 10000 到 100000 倍。

## 评价指标

$$\text{AMAT} = P_{\text{Hit}} \times T_M + P_{\text{Miss}} \times T_D$$

平均内存访问时间（AMAT）= 命中率 × 内存延迟 + 未命中率 × 磁盘延迟。因为磁盘太慢，即使很小的未命中率也会严重拉高 AMAT。

## 几种策略

**最优策略（OPT/MIN）**：踢掉未来最久不会被访问的页。理论最优但不可实现——需要预知未来，只用来做比较基准。

**FIFO**：踢掉最早进入内存的页。简单但效果差。存在 **Belady 异常**——增加物理内存反而可能增加缺页次数。

**随机**：随便踢一个。有时比 FIFO 好，但不稳定。

**LRU**：踢掉最近最久未使用的页。利用历史信息近似预测未来。在大多数工作负载下接近最优。不会有 Belady 异常（LRU 属于栈算法）。

## LRU 的近似实现——时钟算法

精确维护 LRU 的访问时间代价太高。实际系统用**时钟算法**（Clock Algorithm）近似：

- 所有页围成一个环形链表，维护一个时钟指针
- 每页有一个**使用位**（use bit / reference bit），页被访问时硬件自动置 1
- 需要替换时，时钟指针扫描：
  - 使用位 = 1 → 清零，跳过
  - 使用位 = 0 → 选中它换出

效果：最近被用过的页使用位为 1，会被跳过（得到"第二次机会"）。长时间没用的页使用位早被清零，会被选中。

优化：优先换出**干净页**（脏位 = 0），因为不用写回磁盘。

## 工作负载与策略选择

- **无局部性**（完全随机访问）：所有策略表现相同
- **80-20 负载**（80% 访问集中在 20% 页面）：LRU 明显优于 FIFO 和随机
- **循环扫描**（依次访问 N 个页，反复循环）：如果 N 大于物理页数，LRU 和 FIFO 都 100% 未命中（最差情况），随机反而更好

## 抖动

如果进程的活跃页面总量超过了物理内存容量，系统不断换入换出，大量时间花在磁盘 I/O 上,叫**抖动**（thrashing）。

解决方案：

- **准入控制**：不允许运行更多进程，让一部分进程的工作集完全放入内存
- **OOM Killer**：直接杀掉占内存最多的进程，释放内存（Linux 的做法）
