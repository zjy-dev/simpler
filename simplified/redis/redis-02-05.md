# 事务

Redis 事务把多条命令打包，按顺序一口气执行完，中间不会被别的客户端插队。

## 基本流程

```text
MULTI       → 开启事务，后续命令入队
命令1       → 入队
命令2       → 入队
...
EXEC        → 按顺序执行所有入队命令
```

客户端有 `REDIS_MULTI` 标志位表示当前处于事务状态。除了 `EXEC`、`DISCARD`、`WATCH`、`MULTI` 四条命令外，其他命令全部放进事务队列。

## WATCH — 乐观锁

`WATCH` 在 `MULTI` 之前调用，监视任意数量的键。如果被监视的键在 `EXEC` 之前被别的客户端修改了，事务拒绝执行，`EXEC` 返回空。

实现：每个数据库维护 `watched_keys` 字典，键→监视该键的客户端链表。任何对数据库键的修改操作都会遍历 `watched_keys`，把相关客户端的 `REDIS_DIRTY_CAS` 标志打开。`EXEC` 时检查此标志，若已打开则拒绝执行。

## ACID 分析

| 特性 | 支持情况 |
|------|----------|
| 原子性 | 命令要么全入队要么不入队。但单条命令执行出错不会回滚，已成功的命令不撤销 |
| 一致性 | 入队错误拒绝执行；运行错误不影响正确命令的结果；宕机后靠 RDB/AOF 恢复一致 |
| 隔离性 | 单线程串行执行，天然隔离 |
| 持久性 | 取决于持久化配置。无持久化或 RDB 模式下不保证；AOF always 模式下有持久性 |

Redis 事务没有回滚。官方的态度是：事务里出错多半是你代码写错了，不该在运行时才发现，所以干脆不做回滚，保持实现简单。
