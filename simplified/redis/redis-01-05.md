# 对象系统

Redis 不直接用底层数据结构存数据，而是封装了一层对象系统。每个键值对的键和值都是一个 `redisObject`：

```c
typedef struct redisObject {
    unsigned type:4;     // 类型
    unsigned encoding:4; // 编码（底层数据结构）
    void *ptr;           // 指向底层数据结构的指针
    int refcount;        // 引用计数
    unsigned lru:22;     // 最后访问时间
} robj;
```

## 五种类型和编码

键固定是字符串对象，值可以是以下五种类型。每种类型根据数据量自动选择不同的底层编码：

| 类型 | 编码 | 底层实现 | 使用条件 |
|------|------|----------|----------|
| 字符串 | `int` | long 整数 | 值是整数且能用 long 表示 |
| 字符串 | `embstr` | 一次分配的 SDS | 字符串 <= 39 字节 |
| 字符串 | `raw` | 两次分配的 SDS | 字符串 > 39 字节 |
| 列表 | `ziplist` | 压缩列表 | 元素数 < 512 且每个元素 < 64 字节 |
| 列表 | `linkedlist` | 双端链表 | 超出上述限制 |
| 哈希 | `ziplist` | 压缩列表 | 键值对数 < 512 且键值都 < 64 字节 |
| 哈希 | `hashtable` | 字典 | 超出上述限制 |
| 集合 | `intset` | 整数集合 | 全是整数且元素数 <= 512 |
| 集合 | `hashtable` | 字典（值全为 NULL） | 有非整数元素或超 512 个 |
| 有序集合 | `ziplist` | 压缩列表 | 元素数 < 128 且成员 < 64 字节 |
| 有序集合 | `skiplist` | 跳表 + 字典 | 超出上述限制 |

数据量小时用紧凑结构省内存，数据量大时自动转换为高效结构。转换单向不可逆。

## 字符串对象的三种编码

- **`int`：** 直接把整数值存在 `ptr` 里（把指针当整数用）
- **`embstr`：** `redisObject` 和 SDS 分配在一块连续内存里，只需一次 `malloc`/`free`，缓存友好。`embstr` 是只读的，一旦修改就会转成 `raw`
- **`raw`：** `redisObject` 和 SDS 分开分配，两次 `malloc`

浮点数以字符串形式存储，运算时临时转回浮点数。

## 有序集合为什么同时用跳表和字典

`zset` 结构里同时维护了跳表和字典：

```c
typedef struct zset {
    zskiplist *zsl; // 跳表：支持范围操作（ZRANGE、ZRANK）
    dict *dict;     // 字典：O(1) 按成员查分值（ZSCORE）
} zset;
```

单用字典做不了范围查询（无序），单用跳表查单个成员要 O(log N)。两者共享成员和分值的指针，不会浪费额外内存。

## 内存回收

引用计数：创建时 `refcount=1`，被引用 +1，不再引用 -1，降到 0 就释放。

## 对象共享

Redis 启动时预创建 0~9999 的字符串对象。多个键的值如果是这个范围内的整数，直接共享同一个对象（`refcount` 对应增加）。

不共享字符串对象，因为验证两个字符串是否相同是 O(N)，不划算。

## LRU 空转时长

`lru` 字段记录最后一次访问时间。`OBJECT IDLETIME` 命令可以查看空转时长。

当 `maxmemory` 开启且淘汰策略为 `volatile-lru` 或 `allkeys-lru` 时，空转时间长的键会被优先淘汰。
