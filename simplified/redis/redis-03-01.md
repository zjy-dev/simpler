# 复制

执行 `SLAVEOF` 就能让一台 Redis 变成另一台的从服务器，同步数据。

## 旧版复制（2.8 之前）

只有 `SYNC` 命令，流程：

1. 从服务器发 `SYNC`
2. 主服务器执行 `BGSAVE` 生成 RDB 文件，同时用缓冲区记录此后的写命令
3. 主服务器把 RDB 文件发给从服务器，从服务器载入
4. 主服务器把缓冲区中的写命令发给从服务器执行

之后进入命令传播阶段：主服务器每执行写命令就发一份给从服务器。

**缺陷**：从服务器断线重连后必须重新完整 SYNC，即使只断了几秒。BGSAVE 非常耗资源（CPU、内存、磁盘 I/O、带宽），短暂断线后全量同步浪费严重。

## 新版复制（2.8+，PSYNC）

`PSYNC` 支持两种模式：

- **完整重同步**：和旧版 SYNC 一样，用于首次复制
- **部分重同步**：断线重连后只补发缺失部分

### 部分重同步的三个组件

**复制偏移量（offset）**：主从各自维护。主服务器每传播 N 字节，偏移量加 N；从服务器每收到 N 字节，偏移量加 N。比较两者偏移量就知道有没有断开过。

**复制积压缓冲区**：主服务器维护的固定大小（默认 1MB）FIFO 队列，保存最近传播的命令及其偏移量。从服务器重连时上报自己的 offset，如果 offset+1 的数据还在缓冲区内，就执行部分重同步；否则退回完整重同步。

**服务器运行 ID（runID）**：每个 Redis 实例启动时生成 40 位随机 ID。从服务器记住上次对接的主服务器 runID。重连时比对，runID 一致才有可能部分重同步；不一致（换了主服务器）直接完整重同步。

### PSYNC 决策逻辑

```text
从服务器 → PSYNC ? -1              // 首次复制，请求完整重同步
从服务器 → PSYNC <runID> <offset>  // 断线重连

主服务器回复：
  +FULLRESYNC <runID> <offset>     // 执行完整重同步
  +CONTINUE                        // 执行部分重同步
```

## 命令传播阶段

从服务器默认每秒发 `REPLCONF ACK <offset>` 给主服务器，作用：

- 检测网络连接状态
- 辅助 `min-slaves` 配置（保证最少 N 个从服务器延迟不超过 M 秒才接受写入）
- 检测命令丢失（主服务器对比 offset，发现差异就补发）
